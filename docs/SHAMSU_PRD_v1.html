Part 01 · Core Loop

Record, confirm, ship an API.

Every user is one of two roles: a Creator (owns a recorded flow) or a Buyer (rents a creator's endpoint). A user picks a mode — Non-tech (visual, friendly) or Developer (logs, JSON) — switchable per API. Free tier grants 5 API creation attempts per day (the attempt is the unit, success or not).

The three phases every API passes through
1
Record. User clicks the big RED button (explicit trigger — no passive logging). The extension captures XHR/Fetch, form submissions, and navigation simultaneously. Sessions run 10 min standard, 30 min max.
2
Confirm. A Figma/Notion-style editor where Claude restates the flow in plain English, auto-detects variables ("Make 'Bangkok' a variable so callers can search any city?"), auto-suggests output fields, and validates logic. This step is where the API is actually defined — not a cosmetic review.
3
Ship. A real REST endpoint is generated: shamsu.com/api/v1/{api-id}/call, with Bearer-key auth, multiple revocable keys, a JSON spec, and an instruction file.
What the user gets Multi-step flows with if/else branching, versioning (keep or overwrite each re-recording), and an endpoint their own software can call forever.
Design invariant The confirmation editor is the product's beating heart. Everything downstream — replay, pricing tier, reliability — is decided by how well Claude classifies the flow here.

Part 03 · Architecture

The replay engine is the whole ballgame.

Shamsu stores both raw network calls and interpreted DOM steps. The critical decision — locked — is how replay is routed. Defaulting to "always spin up Chrome" is the trap that destroys unit economics at scale. Instead, Claude classifies each API at record-time and routes at call-time.

replay_mode — classified once, routed every call
network_replay LIGHT

Pure data extraction where the payload already lives in a captured XHR/JSON response. Replays the HTTP call directly — no browser. 10–50× cheaper and survives HTML redesigns.

browser_replay MEDIUM / HEAVY

Form submission, login, any stateful or JS-dependent action (CSRF, cookies, validation, anti-bot). Full Playwright DOM replay with credential injection.

hybrid HEAVY

Authenticated extraction ("my booking history"). Playwright establishes the logged-in session; data is pulled from the intercepted network response mid-flight.

Phase 2 — self_heal A network_replay API whose response signature changes auto-falls-back to browser_replay and notifies the creator to re-record. Deferred for MVP — but keep replay_mode extensible.

The call, end to end

POST /api/v1/{api-id}/call
# request
Authorization: Bearer {api-key}
{ "city": "Bangkok", "hotel_type": "luxury" }

# response — standard wrapper
{
  "success": true,
  "data": { "hotels": [ { "name":"Hotel A", "price":100, "rating":4.5 } ] },
  "meta": { "execution_time_ms":3400, "calls_remaining":950 }
}

Flow: validate key → check credits, rate limit, concurrency → deduct cost → branch on replay_mode → execute (navigate, inject variables, submit) → extract output → log & return. Credentials support three per-API modes: stored, caller-provided, prompt-on-call. Users always authenticate with their own credentials; a login failure prompts the creator to re-authenticate in real time.

Part 04 · Operations

Guardrails that keep the platform trustworthy.

Monitoring

An admin dashboard for the Kopa Shamsu team surfaces active APIs, failure rates, response times, Chrome-pool and queue depth, and credential anomalies. Alert levels are Info + Warning only — never Critical. Creators see their own API failures; Shamsu sees which API failed, for which buyer, when, and why.

Abuse prevention — hybrid

  • Automated heuristics — phishing blocklist, password/card extraction, credential-exfil patterns, spam, thin listings.
  • Community reporting — buyers flag suspect APIs.
  • Team review — flagged APIs held before marketplace publish.
Three-stage creator enforcement
Escalation

Warning → Final Warning → Ban. Admin can flag any API and ban any creator from a single console.

Disputes

Buyer opens a ticket → Shamsu auto-tests the API → 50% refund on genuine failure; full refund on fake listings.

Payments & recovery

bKash confirmationManual — support matches the transaction number → confirms → credits auto-added (30–60 min window)
Failed mid-callCharge 50% (infra was spent) · notify creator · caller may retry
Flow backupCreator exports JSON and Shamsu keeps an encrypted library copy — access is never lost
Credential rotation90-day auto-invalidate · 60-day reminder · notify creator
SLA target 99.99% uptime (~52 min/yr) · <5s ideal response, <10s acceptable · graceful degradation puts the marketplace read-only and queues calls rather than dropping them.
Part 05 · Business Model

Simple now. Margin-safe at launch.

Decision locked: universal flat pricing for the beta, weighted tiers at public launch — set from real usage data, not guesswork. Flat-forever loses money on heavy APIs because Chrome + Claude cost scales with complexity.

MVP / Beta

Universal flat — 150 BDT per 10,000 calls. One number, ships fast, unblocks the build. Meter every call from day one.

At launch — Phase 2

Weighted tiers mapped to replay_mode. Margin protected on every call. Creator sees the price after confirmation.

Heavybrowser_replay / hybrid6+ steps, login, >30s~1,000 BDT/10k

Marketplace & audience

  • Split: 60% creator / 40% Shamsu. Creators choose per-call or monthly subscription pricing.
  • First segment: solo online coaches priced out of premium SaaS who need custom APIs.
  • Creator incentive: no code, no expertise — Claude-assisted creation plus a real revenue share.
